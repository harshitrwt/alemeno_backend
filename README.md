# Alemeno Backend

## Overview

This project is a backend system that accepts a files of financial transactions, processes it in the background using a job queue, and produces a cleaned dataset along with flagged anomalies and an automatically generated summary. The system is built around four services that work together: an API for accepting uploads and serving results, a worker that performs the actual processing, a queue that connects the two, and a database that stores everything.

The core idea behind the design is that uploading a file should never make the caller wait. The API accepts the file, hands the work off to a background worker, and responds immediately with a job identifier. The caller can then poll for status until the job finishes and retrieve the full results once it is done.

## Tech Stack

* FastAPI as the web framework for the API
* Celery as the task queue framework for background processing
* Redis as the message broker between the API and the worker
* PostgreSQL as the database for jobs, transactions, and summaries
* SQLAlchemy as the object relational mapper
* Pandas for reading and cleaning the CSV data
* Groq as the large language model provider, used for transaction classification and narrative generation
* Docker and Docker Compose for running the entire system as a set of containers

## Architecture

There are four containers in total.

1. api, which runs the FastAPI application and exposes the endpoints
2. worker, which runs a Celery worker and performs the cleaning, anomaly detection, classification, and summary generation
3. redis, which acts purely as the message queue between the api and worker containers
4. postgres, which stores all persistent data


## Requirements

* Docker and Docker Compose installed on your machine
* A free Groq API key

## Getting a Groq API Key

Groq offers a free tier that is sufficient for running this project. To get a key, visit console.groq.com and generate an API key from the dashboard. You will need this key before running the system.

## Setting Up the Project

1. Clone the project.
2. Run command
   ```
     pip install requirements.txt
   ```

3. Copy the file named .env.example to a new file named .env
4. Open .env and replace the placeholder value for GROQ_API_KEY with your actual Groq API key

## Running the Project

The entire system starts with a single command, run from the root of the project.

```bash
docker compose up --build
```

This single command builds the images and starts all four containers together. Once it finishes starting, the API will be available at:

```
http://localhost:8000
```

## Trying the API

The simplest way to try the API is through the interactive documentation that FastAPI generates automatically. Open the following address in your browser.

```
http://localhost:8000/docs
```

From this page you can upload a CSV file, check the status of a job, and retrieve results, all without needing a separate tool. Each endpoint can be expanded, filled in, and executed directly from this page.

If you prefer using curl from the command line, examples are included below.

## API Endpoints

### POST /jobs/upload

Accepts a CSV file upload, validates it, creates a job record with a status of pending, hands the file off to the worker, and returns the job identifier immediately.

```bash
curl -X POST http://localhost:8000/jobs/upload -F "file=@transactions.csv"
```

### GET /jobs/{job_id}/status

Returns the current status of a job, which will be one of pending, processing, completed, or failed. Once the job is completed, the response also includes a short summary with high level statistics.

```bash
curl http://localhost:8000/jobs/1/status
```

### GET /jobs/{job_id}/results

Returns the full output of a completed job, including the cleaned list of transactions, the flagged anomalies, a per category spending breakdown, and a narrative summary generated by the language model.

```bash
curl http://localhost:8000/jobs/1/results
```

### GET /jobs

Returns a list of all jobs along with their filename, status, row counts, and creation time. This endpoint can be filtered by status.

```bash
curl http://localhost:8000/jobs
curl "http://localhost:8000/jobs?status=completed"
```

## How the Pipeline Works

When a CSV file is uploaded, the following happens in order.

1. The file is saved to disk and a job record is created in Postgres with a status of pending
2. A task is placed on the Redis queue and the API responds immediately with the job identifier
3. The Celery worker picks up the task and reads the file using Pandas
4. The data is cleaned. Dates are converted to a single consistent format, currency symbols are removed from amounts, status values are made consistent in case, and missing categories are filled in
5. Exact duplicate rows are removed
6. Each transaction is checked against two anomaly rules. The first flags any transaction that is far larger than what is typical for that account. The second flags transactions billed in US dollars to merchants that only operate domestically, such as Swiggy, Ola, IRCTC, and Zomato
7. Transactions that still do not have a category are sent to Groq in a single batched request for classification, rather than one request per transaction
8. A second request is sent to Groq asking for a short narrative summary of the spending pattern, along with a risk level of low, medium, or high
9. Both LLM calls are retried up to three times with increasing delays if they fail. If all retries are exhausted, the affected rows are marked accordingly and the job continues rather than failing outright
10. The cleaned transactions and the summary are written to Postgres, and the job status is updated to completed
